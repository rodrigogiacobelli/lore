# ADR-008: CLI `--help` is the primary teaching interface for AI agents

**Status:** ACCEPTED

## Context

Lore is used by AI agents. Those agents must understand the tool before they
can use it correctly. Until now, that understanding was delivered through two
external documents: `AGENTS.md` (generated by `lore init`, instructs the
orchestrator on workflow and dispatch) and the codex (internal Lore
documentation, available only when Lore is used to develop Lore itself).

This creates a gap: an agent working on any project other than Lore has no
codex. It has only the CLI and `AGENTS.md`. If those two surfaces are
insufficient to understand the tool, the agent will misuse it.

`AGENTS.md` is a hand-maintained document. It can drift from the
implementation. It must be updated manually when the tool changes. And it
describes workflow conventions — not the tool's own model.

`--help` is different. It is embedded in the code. It cannot drift from the
implementation. It is always available, always current, and requires zero
extra tool calls beyond those the agent was already making.

Today, Lore's `--help` output describes syntax only: flag names, argument
types, and one-line summaries. An agent reading `lore missions --help` learns
the flags but not what a mission is, how it relates to a quest, or what the
type field means.

## Decision

Lore's `--help` output must teach agents how the tool works, not just how to
call it.

**Scope of enrichment:**

- **Top-level `lore --help`**: A short conceptual orientation paragraph
  before the command list. Explains the core entities (quests, missions,
  knights, doctrines) and the basic workflow (orchestrator creates and
  dispatches, workers claim and close). An agent with no prior knowledge of
  Lore should understand the mental model after reading this.

- **Command-group `--help`** (e.g. `lore missions --help`, `lore knight
  --help`): One paragraph explaining the concept the group operates on —
  what the entity is, why it exists, and when to use these commands.

- **Individual subcommand and flag help**: Not enriched. Flag names and
  argument types are sufficient at this level. Keeping this layer terse
  respects the "minimise tool calls" principle (ADR-001) — agents read
  top-level and group-level help once, then call specific commands directly.

**JSON `--help`:**

`--help --json` is not implemented. Click processes `--help` before any command
handler runs; a working implementation would require patching Click internals
or replacing Click's `--help` option entirely. The implementation cost
outweighs the benefit: agents needing structured CLI reference can use
`lore codex show tech-cli-commands` instead. This is deferred indefinitely.

**AGENTS.md:**

`AGENTS.md` is reduced to a lightweight basics guide. Its role is:
1. Tell the agent that Lore is installed and available.
2. Point to `lore --help` as the authoritative reference for the tool's model.
3. Describe the orchestrator-specific workflow conventions that `--help` does
   not cover (dispatch loop, mission types, claim/done lifecycle).

`AGENTS.md` is not the place to document what a mission is or how the CLI
works. That belongs in `--help`.

## Rationale

**`--help` cannot drift.** It is in the source. Any change to the CLI is
visible in `--help` immediately. `AGENTS.md` requires a separate update step
that is easy to miss.

**Agents on non-Lore projects have no codex.** Lore's own codex documents the
tool in depth, but that codex is only available when Lore is used to manage
Lore development. A consumer project's agents have only the CLI and
`AGENTS.md`. Enriched `--help` removes the dependency on external
documentation for basic understanding.

**Top-level + group-level enrichment is consistent with ADR-001.** ADR-001
requires minimising tool calls. An agent that reads `lore --help` once and
understands the model needs fewer total calls over its lifetime. Enriching
every flag would increase the cost of each `--help` invocation without
proportional benefit. The two-level approach (top + group) gives full
conceptual coverage at minimal verbosity.

**`--help` is maintained independently from the codex.** The codex documents
design decisions, internals, and architecture. `--help` documents usage.
They serve different readers at different levels of depth and are not required
to stay in sync word-for-word.

## Consequences

- `cli.py` is updated with enriched help strings at the top-level group and
  each command-group level.
- `--help --json` is not implemented. Agents needing structured CLI reference
  use `lore codex show tech-cli-commands`.
- The `AGENTS.md` default template (`src/lore/defaults/`) is reduced to
  ~40–50 lines, retaining only the dispatch loop and worker workflow.
- The `board` command group is excluded from this round of enrichment. It does
  not yet exist in the codebase (`spec-mission-board` is unimplemented).
  Enrichment for `board` is deferred until that feature ships.
- Writers of new command groups must include a concept paragraph in the group
  `--help`. This is a new convention for CLI contributions.

## Alternatives considered

**Keep AGENTS.md as the primary reference, enrich it instead.** Rejected.
`AGENTS.md` drifts. It is not the right surface for tool documentation.
Enriching it makes the drift problem worse by adding more content to maintain.

**Reference codex IDs from `--help`.** Rejected. Consumer projects have no
Lore codex. `--help` must be self-contained.

**Enrich all flag-level help.** Rejected. Inconsistent with ADR-001
(minimise tool calls). Flag-level verbosity increases the cost of every
`--help` call without meaningful benefit over top-level + group-level
coverage.
