# Validators Module Internals

**Source module:** `src/lore/validators.py`

This module provides the foundation validation layer for Lore. It has zero imports
from any `lore.*` module and is callable by any layer without circular dependency risk.

## Module Contract

- **Zero `lore.*` imports.** `validators.py` is a pure utility module. Any layer may
  import it without introducing a circular dependency.
- **Pure functions.** All functions are stateless. No side effects.
- **Return contract.** All validation functions return `str | None` — an error message
  string if the value is invalid, or `None` if it is valid.

## Compiled Patterns

| Pattern name | Regex | Purpose |
|---|---|---|
| `_QUEST_RE` | `^q-[0-9a-f]{4,6}$` | Strict quest ID format (hex chars only) |
| `_STANDALONE_MISSION_RE` | `^m-[0-9a-f]{4,6}$` | Standalone mission ID (no quest prefix, hex chars only) |
| `_SCOPED_MISSION_RE` | `^q-[0-9a-f]{4,6}/m-[0-9a-f]{4,6}$` | Quest-scoped mission ID (hex chars only) |
| `_ENTITY_RE` | (union of `_QUEST_RE`, `_SCOPED_MISSION_RE`, `_STANDALONE_MISSION_RE`) | Any valid entity ID |
| `_MISSION_RE` | (union of `_SCOPED_MISSION_RE`, `_STANDALONE_MISSION_RE`) | Any valid mission ID |

All strict patterns use the character class `[0-9a-f]{4,6}` — lowercase hex only, 4–6
characters. IDs generated by `lore.ids.generate_id()` always conform to this constraint.

## Function Reference

| Function | Signature | Accepts | Rejects | Callers |
|---|---|---|---|---|
| `validate_message` | `(message: str) -> str \| None` | Any non-empty, non-whitespace-only string | Empty string, whitespace-only string | `db.add_board_message`, `cli.py` board_add handler |
| `validate_entity_id` | `(eid: str) -> str \| None` | Valid quest ID or any valid mission ID | Anything else | `db.add_board_message`, `cli.py` board_add handler |
| `validate_mission_id` | `(mid: str) -> str \| None` | Scoped or standalone mission ID (strict hex) | Quest IDs, malformed strings | `db.claim_mission`, `cli.py` claim/done/block handlers |
| `validate_priority` | `(priority: int \| None) -> str \| None` | `None` or integer 0–4 | Integer outside 0–4 | `db.create_quest`, `db.create_mission`, `db.edit_quest`, `db.edit_mission`, `cli.py` |
| `validate_name` | `(name: str) -> str \| None` | Alphanumeric start; alphanumeric, hyphens, underscores only | Empty, starts with non-alphanumeric, contains disallowed chars | `cli.py` knight_new, doctrine_new |
| `validate_quest_id_loose` | `(qid: str) -> str \| None` | `^q-[a-z0-9]{4,8}$` — see restricted use below | Anything not matching the loose pattern | Test harness only — see below |
| `route_entity` | `(eid: str) -> tuple[str, str]` | (not a validator — raises on unknown) | — | `db.add_board_message` (internal routing) |

## `validate_quest_id_loose`

This function uses the loose pattern `^q-[a-z0-9]{4,8}$` instead of the strict hex
pattern. The loose pattern accepts non-hex characters (`g`–`z`) deliberately, because
test databases may insert IDs that do not conform to the standard hex format.

**Restricted use:** `validate_quest_id_loose` must not be used for new ID creation or
for user-facing format validation. Its only legitimate use is in paths that must accept
test-DB-inserted IDs (e.g., certain CLI helpers that are exercised with synthetic IDs
in tests). Calling this function for real user input is a bug — use `validate_entity_id`
or `validate_mission_id` instead.

## `route_entity`

`route_entity(eid)` returns `(table, id_col)` for dependency routing. Given a valid
entity ID, it returns:

- `("missions", "id")` — for any mission ID (scoped or standalone)
- `("quests", "id")` — for any quest ID

This function assumes `eid` has already passed `validate_entity_id()`. If called with
an unrecognised ID, it raises `ValueError`. It is only called internally by
`db.add_board_message` after the format validation guard.

## Two-Layer Validation Model

Lore enforces validation at two layers:

1. **`db.py` — authoritative enforcement layer.** All write operations that are
   Realm-callable call `validators.*` before any database access. A malformed or
   invalid value always produces an error return dict before the DB is touched.

2. **`cli.py` — UX translation layer.** CLI handlers call the same `validators.*`
   functions for early exit and click-formatted error messages. These are thin
   translators — they contain no inline regex and no rules of their own. They
   delegate entirely to `validators.*` and format the result for terminal output.

Neither layer duplicates the other's logic. A validation rule lives exactly once —
in `validators.py`.

## Exception: `remove_dependency`

`db.remove_dependency` does not call `validate_mission_id`. Its contract is
existence-based: supplying a malformed ID produces a "not found" response, which
is acceptable for a delete-by-existence operation. This exception is documented in
decisions-011-api-parity-with-cli (lore codex show decisions-011-api-parity-with-cli).
