# AGENTS.md Specification

## Overview

The `AGENTS.md` generated by `lore init` is a first-class deliverable of this project. It is a single file placed at the project root, read by all agents — both orchestrator and workers. Since Claude Code gives every agent (including child agents) access to `AGENTS.md`, it is the lightweight entry point for Lore — directing agents to `lore --help` for entity and CLI model understanding, and capturing only the orchestrator workflow conventions that `--help` does not cover.

`AGENTS.md` is approximately 40–50 lines. It does not reproduce `lore --help` output or document CLI syntax. That content is owned by enriched `--help` (per ADR-008). `AGENTS.md` owns the orchestrator dispatch decision logic (which mission type triggers which action) because that is a workflow convention, not a CLI feature.

## File Structure

The generated file uses HTML comment markers so `lore init` can detect and update its own section without disturbing user-written content:

```markdown
<!-- lore:begin -->
# Lore Task Manager

## For the Orchestrator (Main Agent)

[Orchestrator instructions here...]

## For Workers (Sub-Agents)

[Worker instructions here...]

## General

[Shared guidance here...]
<!-- lore:end -->
```

The content between `<!-- lore:begin -->` and `<!-- lore:end -->` is the Lore-managed section. Everything outside the markers is user content. `lore init` never modifies content outside these markers.

## Required Content

### Section 1: Orchestrator Instructions

Must cover exactly three items:

1. **Tool availability** — Confirm that Lore is installed and direct to `lore --help` for the full entity and CLI model. Do not reproduce `lore --help` output or document quest/mission creation syntax here.

2. **Dispatch loop** — The claim/ready/dispatch/done cycle with type-based branching:
   - `lore ready` returns the next mission with its type
   - Type `knight` → claim and spawn a worker agent with the mission ID
   - Type `constable` → claim and handle inline (orchestrator chore)
   - Type `human` → do NOT claim; leave for the human to handle
   - Pass the mission ID to the worker so it knows what to work on

3. **Mission description requirements** — Descriptions must be thorough, self-contained, and include acceptance criteria, constraints, and relevant file paths. A worker must be able to execute the mission using only the description.

All other orchestrator guidance (quest/mission creation syntax, knight assignment, monitoring, codex access, artifact access) is covered by enriched `lore --help` and command-group help pages.

### Section 2: Worker Instructions

Must cover:

- The worker has been assigned a specific mission ID by the orchestrator
- How to get all context in a single call via `lore show <mission-id>` (returns Mission details and Knight contents together)
- How to mark work done via `lore done` when the Mission is complete (cascades automatically)
- How to report blockers via `lore block` with a clear reason
- What **not** to do: workers do not create quests or missions. They do not claim work. They execute and close.

## Update Behaviour on Re-init

On `lore init` (re-run):

- If `AGENTS.md` does not exist: create it with the reduced Lore template (~40–50 lines).
- If `AGENTS.md` exists and contains the marker line `<!-- lore:begin -->`: replace the content between `<!-- lore:begin -->` and `<!-- lore:end -->` with the latest Lore template. Content outside the markers is preserved.
- If `AGENTS.md` exists **without** Lore markers: back it up to `AGENTS.md.old` (using `Path.replace()` for cross-platform safety) and write a fresh Lore `AGENTS.md`. If `AGENTS.md.old` already exists, it is overwritten.

## Template Source

The template content for the Lore-managed section is stored at `src/lore/defaults/AGENTS.md` in the source package. This file is the authoritative source for what the generated `AGENTS.md` contains.
